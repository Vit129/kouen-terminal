## Active Decisions
- [2026-07-08] **`MobileBridgeServer` printed a pairing URL pointing at a page it never served — only the dev script (`Scripts/mobile-web.sh`, a separate `python3 -m http.server`) ever made that URL actually load; production/preview daemons had nothing listening on the page port at all.** Caught by `curl`-ing a real `make preview` daemon's page port right after confirming the WS/session-list side worked — connection refused, even though the printed QR/URL looked completely valid. Classic case of "the feature that only ever ran inside its own dev harness quietly depended on that harness's scaffolding." Fixed by moving the page itself into the daemon (`MobileBridgeServer.embeddedPageHTML` + `makePageListener`, a plain-HTTP NWListener alongside the WS one, same loopback+Tailscale bind scope) — single source of truth, no separate process to forget to spawn. **How to apply:** when a feature's pairing/QR/URL flow was only ever verified through a dev script that also spins up auxiliary infrastructure (a page server, a mock service, etc.), explicitly check whether production/preview has that same infrastructure — don't assume "verified via the dev script" implies "verified end-to-end in the real deployment shape." A quick `curl`/direct-connect against the actual production-path process is cheap insurance.
- [2026-07-08] **A Settings toggle whose effect is read only at daemon-launch time (not live-applied) needs its own explicit restart trigger — the existing build-handshake staleness check (`DaemonLauncher.daemonIsStale`) does not cover it.** Wiring `KouenSettings.mobileBridgeEnabled` into the production daemon: `MobileBridgeServer` only starts by checking `KOUEN_MOBILE_BRIDGE_PORT` once at daemon-process launch (`main.swift:186`), unlike most Settings toggles (`SettingsAdvancedView`'s tmux-style options) which apply live via `OptionStore`+IPC with no restart. Reused `LaunchAgentInstaller.install`'s existing content-diff bootout/rewrite (adding the env var to the generated plist naturally changes `desired` and triggers it) instead of building new IPC start/stop plumbing — restart-required was a deliberate, confirmed trade-off, not an oversight. **How to apply:** before adding a new Settings toggle, check whether the thing it controls is read continuously (OptionStore path, no extra work) or only once at process/daemon start (needs an explicit restart call wired to the toggle's `onChange`, and the DEBUG/preview fallback-process path needs its own force-kill+respawn, not just the launchd path — `spawnFallbackProcess` previously no-op'd if a fallback process was already running, silently swallowing the setting change in DEBUG builds until `forceRestart:` was added).
- [2026-07-08] **Terminal ASCII-QR "too big" complaint: font size / module count shrink the true on-screen footprint; ASCII-packing scheme (half-block vs quadrant-block) does not.** `qrAsciiArt` (`MobileBridgeServer.swift`) went double-width (78x39 chars, confirmed too big) → half-block ▀▄█ (39x17, correctly square — 2 module-rows packed per terminal row, matches terminal chars rendering ~2x taller than wide) → quadrant-block (20x17, packs a 2x2 module block per char) — this last one was a regression: compressing 2 module-*columns* per char on top of the half-block's existing 2-module-*row* compression, with no compensation for the terminal's ~2:1 char height:width ratio, stretched the QR visibly non-square (caught via user screenshot, not caught by any automated check). Reverted to half-block. **How to apply:** if a rendered-QR/ASCII-art "too big" complaint resurfaces, the only real levers are a shorter encoded payload (fewer QR modules) or a smaller font/view scale — don't reach for a denser character-packing scheme expecting it to shrink the actual rendered size; verify any new packing scheme's squareness against terminal char aspect ratio (~1 char-width : ~2 char-height) before shipping, not just its character-count reduction.
- [2026-07-08] **`make mobile-web`'s isolated dev daemon showing zero sessions when paired from a phone is by design, not a bug** — `Scripts/mobile-web.sh` runs a separate `KouenDaemon` process against an isolated `/tmp/kouen-mobile-web-<hash>` `KOUEN_HOME` (own `SurfaceRegistry`), deliberately never the real running Kouen.app's sessions. **How to apply:** don't debug "no sessions found" on this dev path as a code bug — the actual gap is that production `Kouen.app` has no wired path to enable `MobileBridgeServer` on itself (`main.swift:186` only starts it when `KOUEN_MOBILE_BRIDGE_PORT` env var is set, which only this dev script sets); pairing against real in-use sessions needs that wired into the app first (not yet built).
- [2026-07-07] **P25 mobile bridge (`p25-mobile-web-mvp` branch): build-green and unit-tests-passing were NOT sufficient proof the daemon-side WS bridge actually ran — a component with its own background async lifecycle (listeners, timers, loops) needs a real "did it start and do something" check, not just a compile check.** `MobileBridgeServer().start(...)` in `main.swift` was a bare unretained temporary; ARC deallocated it right after `start()` returned, silently turning every `[weak self]` closure inside it (listener callbacks, the pairing loop) into a no-op — the daemon logged "listening on..." and then did nothing else, no error, because a deallocated `weak self` failing a guard isn't an error, it's silence. Two earlier slices had already reported "verified: build clean, robot tests pass" before this was caught — that report was true but incomplete. Found only because a live protocol test (a Python `websockets` client acting as the phone, then re-verified against a real `make preview`-spawned GUI daemon) was run against it. **How to apply:** for any new daemon-side component with async listeners/background loops, don't stop at build+unit-test green — do at least one live round-trip against a real running instance before calling it done, and retain any object whose closures matter for the process's actual lifetime (top-level `nonisolated(unsafe) var`, same pattern as the existing `signalSources` array), never a bare expression-statement temporary.
- [2026-07-07] **`DaemonLauncher.spawnFallbackProcess` (the GUI app's real daemon-spawn path) sets `proc.standardOutput = nil`/`standardError = nil` — anything a daemon-side feature only `print()`s to stdout is invisible in the actual shipped app, only visible when someone manually runs the `KouenDaemon` binary in a foreground terminal.** Caught because the P25 mobile-bridge pairing QR/URL was plain `print()`-only; worked fine in the dev-only `make mobile-web` flow but reached daemon.log with zero entries when tested through the real GUI-spawned daemon (`make preview`). **How to apply:** any daemon-side message meant to survive into the real production deployment shape must go through `daemonLog()` (writes to `daemon.log`, survives the nil'd stdio), not plain `print()` — reserve `print()` for genuinely interactive dev-only tooling (e.g. a script meant to only ever be run in a foreground terminal).
- [2026-07-06] **Nested cross-origin iframe wheel-scroll broken in browser pane — fix CONFIRMED (repeated-scroll retest passed, `moved=true` persisted).** A claude.ai artifact URL (content in a nested iframe at `<uuid>.frame.claudeusercontent.com`) won't wheel-scroll in Kouen's browser pane; scrollbar drag works, ordinary pages wheel-scroll fine, and Safari scrolls the same artifact URL fine — Kouen-specific. Root cause (Opus subagent investigation): WebKit's async scrolling thread never builds a scrolling-tree node for the nested iframe at initial layout (matches WebKit bugzilla 124139); scrollbar drag works via a different code path (web process `RenderLayer`, not the scrolling thread). A magnification change forces the tree commit that builds the node — confirmed by the user's manual pinch-to-zoom making scroll start working. Fix in `BrowserPaneView.swift`: `kickCompositorRelayout(for:)` nudges `webView.magnification` by `+0.01` then reverts 50ms later, triggered by an in-frame script posting on the nested frame's first `pointermove`/`wheel` (not a blind post-load timer — an earlier `didFinish + 1.2s` version fired before the iframe had mounted). **Known unresolved risk (flagged by the investigation itself, not yet disproven):** the revert-to-original step re-runs the tree build at the old scale — if the node is fragile this could re-drop it, reproducing an earlier "scrolled a little then stopped" symptom via a different mechanism. A `#if DEBUG`-gated probe logs `scrollTop` before/after each nested-frame wheel event to the existing `kouenConsoleLog` file pipe. **How to apply:** confirmed via repeated-scroll manual retest (multiple gestures over several seconds, not just one) — the revert-to-original step did not re-drop the scroll-tree node. If this ever regresses, the named fallback is to hold the nudged magnification instead of reverting. Full ledger (all ruled-out candidates: magnification/back-forward-gesture toggles, UA spoofing, native AppKit event interception, capture- vs bubble-phase wheel logging) in `knowledge/ui/browser-pane.md`.
- [2026-07-06] **P35 fixed: Google OAuth login inside the browser pane failed at the final "Allow" step because `BrowserPaneView.createTab` called `.load()` on the popup webview after WebKit had already created it via `createWebViewWith` — a redundant, disconnected navigation that severed `window.opener`.** Original plan doc guessed the cause was Google's anti-phishing embedded-webview block (real, but not what fired here — repro reached the consent screen fine, proving the popup wasn't flagged). Live repro with `NSLog`/injected `console.log` diagnostics (`window.opener` check + `message` event listener) showed `opener=null` at every navigation inside the Google popup, even the very first one — ruling out COOP (which would also break real Safari) and confirming a local bug. Per Apple's documented contract, `webView(_:createWebViewWith:for:windowFeatures:)` must return the created `WKWebView` *without* loading anything into it — WebKit auto-loads `navigationAction.request`. Fix: `createTab(url:configuration:skipLoad:)` gets a `skipLoad` param, `true` only on the `createWebViewWith` path; also added the missing `webViewDidClose` (JS `window.close()` was a silent no-op, leaving Google's `gsi/transform` relay tab orphaned even after the opener fix). Verified end-to-end: Google login → Allow → popup auto-closes → claude.ai artifact loads authenticated. **How to apply:** any future `WKUIDelegate.createWebViewWith` implementation must never call `.load()`/`.loadFileURL()` on the view it returns — treat this as a hard WKWebView invariant, not a one-off. `knowledge/ui/browser-pane.md` has the full case + diagnostic technique (temp `window.opener`/`message`-event console.log injection reusing the existing `kouenConsoleLog` pipe, since page JS console output lands in a `/tmp` file, not app stdout).
- [2026-07-06] **P36 (app icon light/dark) shipped both: light recolor (`#93a889`→`#567a52`, fixed a white-edge SVG-letterbox bug) + dark Dock icon via runtime `NSApp.applicationIconImage` swap — NOT via any static build mechanism.** Confirmed empirically that macOS `"idiom":"mac"` app icons never supported per-appearance appiconset variants (built + compiled one via `actool` to test; `assetutil` on the resulting `Assets.car` showed 0 of 10 dark renditions made it in — actool silently orphans them as "unassigned children"; that mechanism is iOS-only). Icon Composer's `.icon` (the real Tahoe-native path) stays blocked — GUI-only, undocumented `icon.json`. Real fix: `AppDelegate.updateDockIconForCurrentAppearance()` sets `NSApp.applicationIconImage` from the existing `effectiveAppearance` KVO observer (same one already driving terminal theme auto-switch) + `NSApp.dockTile.display()`; only swaps the *running* app's Dock tile, Finder/Launchpad/Get-Info stay static (read before the process exists) — inherent to the technique, not a bug. **Lesson: `NSRunningApplication.icon` queried from an external process does NOT reflect a target app's live `applicationIconImage` change — it's stale/cached. Don't use it to verify; trust an NSLog trace of the setter, or check the real Dock.** Full case in `agent-memory/plans/p36-app-use-white-dark-auto-mode.md`.
