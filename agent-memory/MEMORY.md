- [2026-07-06] **P35 fixed: Google OAuth login inside the browser pane failed at the final "Allow" step because `BrowserPaneView.createTab` called `.load()` on the popup webview after WebKit had already created it via `createWebViewWith` — a redundant, disconnected navigation that severed `window.opener`.** Original plan doc guessed the cause was Google's anti-phishing embedded-webview block (real, but not what fired here — repro reached the consent screen fine, proving the popup wasn't flagged). Live repro with `NSLog`/injected `console.log` diagnostics (`window.opener` check + `message` event listener) showed `opener=null` at every navigation inside the Google popup, even the very first one — ruling out COOP (which would also break real Safari) and confirming a local bug. Per Apple's documented contract, `webView(_:createWebViewWith:for:windowFeatures:)` must return the created `WKWebView` *without* loading anything into it — WebKit auto-loads `navigationAction.request`. Fix: `createTab(url:configuration:skipLoad:)` gets a `skipLoad` param, `true` only on the `createWebViewWith` path; also added the missing `webViewDidClose` (JS `window.close()` was a silent no-op, leaving Google's `gsi/transform` relay tab orphaned even after the opener fix). Verified end-to-end: Google login → Allow → popup auto-closes → claude.ai artifact loads authenticated. **How to apply:** any future `WKUIDelegate.createWebViewWith` implementation must never call `.load()`/`.loadFileURL()` on the view it returns — treat this as a hard WKWebView invariant, not a one-off. `knowledge/ui/browser-pane.md` has the full case + diagnostic technique (temp `window.opener`/`message`-event console.log injection reusing the existing `kouenConsoleLog` pipe, since page JS console output lands in a `/tmp` file, not app stdout).

- [2026-07-02] **P34 F2/F3: MCP `harnessGetLastBlock`/`harnessGetBlock` needed no new daemon-side OSC-133 parser — `RealPty.captureGrid` already replays retained scrollback through a fresh `HarnessGridTerminal` on every call, so `RealPty.block(id:)` just does the same replay and reads the block store.** The daemon itself never parses OSC 133 live (it's a dumb byte-relay + raw scrollback store; only client-side `TerminalEmulator` instances — GUI surface, `harness attach` — parse it), which looked like a real architecture gap for F3 until this precedent surfaced. Fix: flipped `documentView` subclass (`override var isFlipped: Bool { true }`), same pattern as `GitPanelView.swift`'s existing `FlippedView`. **Lesson: when several small layout nudges only partially converge, stop nudging and check `isFlipped` before adding another.** See `knowledge/cases/appkit-ui.md` CASE-059, `knowledge/rl-lessons.md` RL-059.

- [2026-07-02] **A blanket `.git/`-path FSEvent filter in `GitPanelView.swift` swallowed external `git commit`/`push` refreshes, not just its own auto-stage noise.** `f6ffb0a` (fixing an FSEvent storm from the panel's own `git add` during auto-stage) filtered out *every* event under `.git/`. But a plain terminal `git commit`/`push` only touches paths under `.git/` (`index`, `COMMIT_EDITMSG`, `logs/HEAD`, `refs/**`) — no working-tree files — so the panel silently stopped seeing those too, and stayed on stale status indefinitely. `suppressingFSEvents` already guarded the panel's own write separately, making the blanket filter redundant. Fix: narrowed to `nonisolated static func isNoisyGitInternalPath(_:)` — only `.git/index`, `.git/index.lock`, `.git/objects/**` count as noise. **Lesson: a filter meant to stop a self-triggering loop should target the specific write that causes the loop, not the whole path prefix it lives under** — the prefix is usually shared with legitimate external signals. Regression test: `Tests/HarnessAppTests/GitPanelViewFSEventFilterTests.swift`.
