# Memory — harness-terminal

## Active Decisions
- [2026-07-02] **Two unrelated file-preview bugs shared no root cause despite sounding like one report — split via `AskUserQuestion` before touching code.** Bug A: `SyntaxTextView.load()`/`applyDiagnosticAttributes()` both do a full `textView.textStorage?.setAttributedString(...)` replace (on `FileChangeWatcher` reload — fires on ANY fs event incl. bare `.attrib` touch, 300ms debounced — and on async LSP diagnostics push); confirmed via a standalone `swift` script repro that this collapses `NSTextView.selectedRange()` to `{length,0}` even with byte-identical text, while scrolling alone never touches selection (disproved "scroll" as the cause). Fix: `preservingSelection(_:)` helper captures/restores `selectedRange()` around both call sites. Bug B: `URLDetection.detectFilePath`'s unquoted-token fallback treated only whitespace/quotes as boundaries, so a coding-agent CLI's paren-wrapped tool-call output (`⏺ Update(path/to/file.swift)`, no space before the path) got swept into one token with `Update(` stuck on the front, which never matched a real file → silent no-op click. Fix: added `(`/`)` to that one boundary-char set. **Lesson: don't assume a two-symptom bug report shares one root cause — ask which concrete UI action fails for each symptom before hypothesizing, since a single fix attempt can look "done" while leaving the other half of the report unaddressed.**
- [2026-07-02] **`IPCRequest.updateTabTitle(surfaceID:title:)` actually mutates `Tab.title` (tab-scoped), NOT `PaneSurface.title`, despite taking a `surfaceID` param — `SessionEditor.updateTabTitle` resolves it via `tabIndex(surfaceID:)` then writes `tabs[tabIndex].title`. `PaneSurface.title` itself was never mutated anywhere in the codebase (vestigial field, default `"Shell"` only).** Discovered while implementing a "pane purpose label" MCP tool — the original backlog note assumed reusing `updateTabTitle` needed no new schema, which was wrong on two counts: wrong granularity (tab, not pane) and gated on the per-tab `automatic-rename` flag (silently no-ops or gets clobbered by the next OSC title update), so it can't serve as a durable per-pane label anyway. Built a dedicated `PaneSurface.label: String?` field + `setPaneLabel` IPC/MCP path instead. **Lesson: a request/field's parameter name (`surfaceID`) doesn't guarantee its actual scope — read the handler's mutation, not just its signature, before reusing an existing IPC path for a new purpose.**
- [2026-07-02] **P34 F2/F3: MCP `harnessGetLastBlock`/`harnessGetBlock` needed no new daemon-side OSC-133 parser — `RealPty.captureGrid` already replays retained scrollback through a fresh `HarnessGridTerminal` on every call, so `RealPty.block(id:)` just does the same replay and reads the block store.** The daemon itself never parses OSC 133 live (it's a dumb byte-relay + raw scrollback store; only client-side `TerminalEmulator` instances — GUI surface, `harness attach` — parse it), which looked like a real architecture gap for F3 until this precedent surfaced. Replaying stored bytes isn't "retroactive backfill" (F1 explicitly rejected that) since the bytes contain the same live `133;C`/`D` sequences originally parsed — deterministic recomputation. `TerminalBlock` promoted back to `public` (was trimmed to internal in F1 since only `commandText` needed it; F2 needed the full shape). **Lesson: before assuming an MCP tool needs new live state tracking, check whether the daemon already has an on-demand replay-through-a-fresh-instance pattern for a sibling feature.**
- [2026-07-02] **P34 F1 slice 1: zsh/fish now emit OSC `133;C;<base64 command>` (their own preexec hook's literal command text) instead of no command-boundary marker at all — bash deferred (DEBUG-trap footgun, `ponytail:` ceiling comment).** Plan doc assumed only a boundary marker was missing; research found `SemanticMark` tracks only `exit: Int?` (no command text ever persisted) and NO shell script emitted `133;B`/`133;C` at all, so the existing `onCommandFinished` duration feature never fired against a real shell. New per-pane `TerminalBlockStore` (`Emulator/TerminalBlock.swift`), decoupled from `HistoryLine` so blocks survive scrollback eviction. `BlockActionBar.rerunBlock()` now reads the real command via `TerminalEmulator.commandText(atPromptLine:)` instead of its old regex prompt-strip (kept as bash-only fallback). Consulted `advisor` before touching the shell scripts — confirmed skipping `133;B` entirely is safe (engine already treats it as C-fallback-only) and avoided embedding a marker in `$PROMPT`/`PS1` (fragile against starship/p10k). **Lesson: when a plan needs a shell script to carry actual data (not just fire a boundary), check whether the shell's own hook already has it (zsh/fish preexec receive the command as an argument) before reaching for screen-scrape/regex.**
- [2026-07-02] **Isolated preview environments start with default settings (`minimumContrast: 3.5`, null `paletteHex`) causing color mismatches against customized production profiles — not a renderer bug.** `make preview`'s isolated state dir (`/tmp/harness-preview-<hash>/settings.json`, fresh per run) doesn't inherit the user's real `~/Library/Application Support/Harness/settings.json` (`minimumContrast: 1` = off, custom 16-color `paletteHex`). Both settings directly recolor terminal output via `CellColorResolver.swift`, so a fresh preview visibly shifts colors (e.g. a CLI's truecolor logo) with zero renderer code differences. Confirmed via `script -q` raw-byte capture: both builds emit byte-identical `38;2;r;g;b` truecolor SGR — the divergence is entirely in settings, not parsing/compositing. Fix: copy production `settings.json` into the preview state dir before comparing colors. See `knowledge/cases/misc.md` CASE-060 for the full elimination path (TERM/COLORTERM/TERM_PROGRAM/terminal-identity/shaderEffect/colorGamut/sampler/SGR-parser/block-compositing all ruled out first).
- [2026-07-02] **`BoardViewController`'s scroll `documentView` was a plain non-flipped `NSView()` — content anchored bottom-left, and `scroll(to: .zero)` scrolled to the bottom, not the top.** Symptom (Sessions→Board switch shows first column header squashed/shifted down) looked like a timing bug and absorbed 5 rounds of nudges (`reload(force:)`, `layoutSubtreeIfNeeded()`, `displayIfNeeded()`, `scrollToTop()`) that only partially helped, because none addressed the actual coordinate-space bug. Fix: flipped `documentView` subclass (`override var isFlipped: Bool { true }`), same pattern as `GitPanelView.swift`'s existing `FlippedView`. **Lesson: when several small layout nudges only partially converge, stop nudging and check `isFlipped` before adding another.** See `knowledge/cases/appkit-ui.md` CASE-059, `knowledge/rl-lessons.md` RL-059.
- [2026-07-02] **File-preview tabs were a single app-wide `FileTabManager` singleton — opening a file in one terminal Tab made it visible in every Tab.** Promoted to `[String: FileTabManager]` keyed by `tabID` on `FilePreviewCoordinator`, swapped via `switchToTab(tabID:)` on every `snapshotChanged` tick (mirrors `PaneLifecycleManager.containerCache`'s tabID-keyed cache+prune pattern exactly, rather than a new design). Confirmed via `interview` skill that scope should be per-Tab (not per-Session, not per-split-pane) and that cross-restart persistence is an explicitly deferred phase-2, not required now. **Lesson: check for an existing per-tabID cache pattern before designing a new per-X scoping mechanism** — `PaneLifecycleManager` already had the exact shape needed (dict keyed by tabID, swap+hide on switch, prune on structural change).
- [2026-07-02] **A blanket `.git/`-path FSEvent filter in `GitPanelView.swift` swallowed external `git commit`/`push` refreshes, not just its own auto-stage noise.** `f6ffb0a` (fixing an FSEvent storm from the panel's own `git add` during auto-stage) filtered out *every* event under `.git/`. But a plain terminal `git commit`/`push` only touches paths under `.git/` (`index`, `COMMIT_EDITMSG`, `logs/HEAD`, `refs/**`) — no working-tree files — so the panel silently stopped seeing those too, and stayed on stale status indefinitely. `suppressingFSEvents` already guarded the panel's own write separately, making the blanket filter redundant. Fix: narrowed to `nonisolated static func isNoisyGitInternalPath(_:)` — only `.git/index`, `.git/index.lock`, `.git/objects/**` count as noise. **Lesson: a filter meant to stop a self-triggering loop should target the specific write that causes the loop, not the whole path prefix it lives under** — the prefix is usually shared with legitimate external signals. Regression test: `Tests/HarnessAppTests/GitPanelViewFSEventFilterTests.swift`.
- [2026-07-02] **Status classification/color must have exactly one source (`BoardModel.columnKind` + `BoardColumnKind.color`) — duplicated copies drift silently.** Uncommitted sidebar work had grown a local `columnKind(for:)` copy in `SidebarListModel.swift` that silently dropped the `agent.activity == .working` check the shared classifier relies on (agents attached to a shell's foreground process with no distinct `currentCommand` would wrongly read Idle), plus two separate hardcoded color-mapping copies in `SidebarSessionListView.swift` (one had `.running`↔`.done` colors swapped vs the canonical `BoardColumnKind.color` in `HarnessDesign.swift`). Always grep for existing `BoardModel.columnKind`/`.color` call sites before writing a new switch over `BoardColumnKind` — the duplication is easy to introduce mid-edit and easy to miss since values often *happen* to match at write time.
- [2026-07-01] **`NSSplitView.setHoldingPriority` does nothing when a classic `constrainMinCoordinate`/`constrainMaxCoordinate` delegate is present — it still redistributes divider position proportionally on window resize.** Sidebar coded at fixed 220pt (`HarnessDesign.sidebarWidth`) via `setPosition(_:ofDividerAt:)` at launch only; any later window resize/restore-to-larger-frame ballooned it proportionally (confirmed via `NSLog`: 480pt window → sidebar 220 correct, then window resized to 1440pt → sidebar 661, same 45.9% ratio — `constrainMinCoordinate`/`Max` (200-320pt) only caps *user drags*, not automatic resize). Fix: implement `NSSplitViewDelegate.splitView(_:shouldAdjustSizeOfSubview:)` returning `false` for the pinned subview — this is the API that actually opts a subview out of auto-resize when a classic delegate is in play. `MainSplitViewController.swift` `SplitChromeDelegate`.
- [2026-06-30] **`applyInitialSidebarState` with `sidebarCollapsedOnLaunch=true` mutates `settings.sidebarVisible` directly without `settings.save()`.** In-memory = `false`, disk = `true`. Not itself the bug (toggle reads in-memory), but means the disk value never reflects launch-forced-collapsed state. If anything reloads from disk during session, toggle direction flips. Consider calling `setSidebarVisible(false, animated: false)` instead to save atomically. (Still-open suggestion, not yet applied.)
- [2026-06-29] **Tab-switch caching invariant:** `detachHostsOnly()` = structural rebuild only. Tab-switch path (`force=false`) MUST NOT call `detachHostsOnly()` before caching — hosts are shared single-instance; detach strips the cached container, fast path reveals empty = black. Always validate `Set(displayNode.allSurfaceIDs()).isSubset(of: cachedHosts.keys)` before fast-path reveal; evict+rebuild on mismatch. See `knowledge/bugs/tab-switch-black-screen.md` (4 FM).
- [2026-06-29] **force=true rebuild must evict cache, not overwrite with empty.** `detachHostsOnly()` followed by `containerCache[prevTabID] = old` stores an empty shell → black on revisit. Fix: `containerCache.removeValue(forKey:) + removeFromSuperview()` when `force=true`. Also: evict existing orphan before `containerCache[tabID] = newContainer` to prevent hidden-Metal-surface leak.
