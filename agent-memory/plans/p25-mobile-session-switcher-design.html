P25 — mobile session switcher

Scan once. Every session is already there.

The earlier spike bound one QR to one terminal — scan again to switch sessions. This design pairs the phone with the whole daemon in a single scan, so switching sessions is a tap, not a re-scan.

01

Scan once, from any screen on the Mac

The QR still opens Safari with a token in the URL — same mechanism as before. What changes is what the token grants.

02

The token pairs with the daemon, not one surface

On connect, the phone gets the live session list over the same socket — titles, working directories, a status pill per session.

03

Tap a session to open it — tap the switcher to leave

Opening a session subscribes on the existing connection; going back unsubscribes. No new pairing, no new token, ever.

04

Tap + to spawn a new session on the Mac, from the phone

Sends a create-session request over the same connection (F6's "create a session on a remote host") — no new pairing, appears in the list like any other session.

Design notes

Try it →
Tap the phone: Scan → Paired → tap a session to open it → tap the ↕ icon in the terminal header to switch without leaving, or tap + in the list/sheet header to spawn a new one. Sample data reuses the three real sessions this spike attached to during testing.
What the wire protocol needs to add
Right now PairingBox binds one token to one surfaceID, and attachToPairedSurface subscribes immediately on auth. This design needs the token to grant the daemon as a whole: on auth, push a {"sessions":[...]} frame (from the real .listSurfaces call, already have this), then accept a client-sent {"attach":"<surfaceID>"} message to subscribe, {"detach"} to drop the current subscription and return to the list, and {"spawn":{"cwd":...}} to create a new session (reuses the real spawnSession daemon call) — all multiplexed over the one already-open connection, no reconnect.
Tradeoff: broader grant per pairing
The current design's whole point was closing "any paired client can land on any surface." This change reopens that on purpose — one scan, deliberately, grants the whole workspace, because that's the actual use case here (Vit's own phone, own Mac, over his own Tailscale). The real requirement it doesn't remove: per-device revocation (F3) still needs to exist before this is anything but a personal spike, since right now there's still no way to revoke a specific paired phone without restarting the whole bridge. Spawning a session from the phone widens this further — same trust boundary as attach, so it rides on whatever F3 solves, not a separate concern.
Resize is still unsolved
Switching sessions mid-connection makes the "one hardcoded 80×30 terminal size" gap from the single-session spike more visible, not less — worth solving alongside this, not after.
