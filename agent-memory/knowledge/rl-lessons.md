# RL Lessons — harness-terminal

Grep target: `grep -n "RL-<number>\|<keyword>" knowledge/rl-lessons.md`

## AppKit / Views

- RL-004: Never reparent Metal terminal surfaces — 1-2s black screen (CASE-003)
- RL-010: `NSView.displayLink` does NOT strongly retain target — always `deinit { displayLink?.invalidate() }`
- RL-021: Pure transparent window fails on bright bg — use `window.backgroundColor = themeColor.withAlphaComponent(opacity)`
- RL-030: Every `snapshotChanged` consumer must check `metadataOnly` flag before rebuilding
- RL-031: Double-subscription — parent routes to child AND child has own observer = fires twice
- RL-032: NSAlert first button auto-gets `keyEquivalent = "\r"` — Enter ALWAYS fires it regardless of Tab focus. Clear it for destructive dialogs.
- RL-033: Borderless NSPanel `canBecomeKey` defaults to `false` — must subclass and override to accept keyboard input.
- RL-035: `local` keyword only works inside bash functions — shell scripts with inline logic must omit it.
- RL-037: `CADisplayLink` does NOT retain target on macOS (only iOS does) — always invalidate in deinit.
- RL-038: `NSPanel` never takes `mainWindow` status — use `NSApp.mainWindow` (not `keyWindow`) to anchor floating panels; on 2nd+ open `keyWindow` points at the panel itself.
- RL-039: Menu `@objc` actions fail before first user click (`keyWindow=nil` on launch). Chain: `keyWindow ?? mainWindow ?? windows.first(where: { $0.contentViewController is MainSplitViewController })`.
- RL-042: `KeyModifiers` name exists in HarnessTerminalEngine — adding same in HarnessCore causes ambiguity. Use `MenuModifiers`.
- RL-043: NSClickGestureRecognizer on parent view intercepts child NSButton clicks — check click location in handler or use mouseUp override instead.
- RL-044: Documenting a verb in COMMANDS.md/BannerShortcutRegistry without adding it to `CommandParser.buildCommand` + `knownVerbs` = silent `unknownCommand`. Always wire both layers.
- RL-049: `nonisolated override func layout()` + `assumeIsolated` is WRONG for non-zombie-prone views (TerminalTabBarView, WindowBorderOverlayView) — triggers thunk executor check crashes under Swift 6.3/6.4. Use standard `override func layout()`.
- RL-050: Retain cycles in NSEvent local monitors prevent deinit. Use `[weak self]`.
- RL-051: `NSTableView.view(atColumn:row:makeIfNecessary:)` throws NSRangeException if row ≥ count. Call `reloadData()` BEFORE iterating rows. Guard with `min(rows.count, sessionTable.numberOfRows)`.
- RL-054: Toggling `isHidden` on NSSplitView subview — always call `split.adjustSubviews()` explicitly to force other subviews to resize.
- RL-067: A file/line reference from an earlier design pass (even your own, from a since-lost session) can point at DEAD code — this app has both an AppKit component (`AgentChipView` in `KouenDesign.swift`) and the actual live SwiftUI render path (`TerminalTabBarView.swift`/`SidebarSessionListView.swift` render the agent icon inline via `Image(nsImage:)`, never touching `AgentChipView`) for the same visual element, and only one is reachable. Grep for real callers (`grep -rln "ClassName"`) before extending a class a design doc names — "the design said this file" is not the same as "this file is on the render path." Caught during P38 Phase B (2026-07-14) after editing `AgentChipView` for a badge that would never have rendered.

## Swift 6 / Concurrency

- RL-034: Zombie surface view — discardCursorRects + resign first responder in `viewWillMove(toWindow:nil)`. Swift 6 nonisolated deinit workaround: `perform(NSSelectorFromString("discardCursorRects"))`.
- RL-036: NotificationCenter `queue: .main` closures are `@Sendable` — wrap body in `MainActor.assumeIsolated` for Swift 6.
- RL-040: `@MainActor @objc` thunks on macOS 26.5/Swift 6.3 call `swift_getObjectType(self)` before method body — zombie = crash. Fix: `nonisolated override` + `MainActor.assumeIsolated` on ALL high-freq AppKit callbacks. See `knowledge/bugs/zombie-crash-macos26.md` for full detail.
- RL-041: `keyUp` arrives in a LATER event loop iteration than `keyDown`. Hold zombie-prone views ≥500ms.
- RL-045: Override `removeFromSuperview()` on zombie-prone views — catches ALL removal paths at single chokepoint.
- RL-046: `nonisolated + assumeIsolated` must cover ALL `@objc` callbacks: `layout()`, `resetCursorRects()`, `viewDidMoveToWindow()`, `viewDidMoveToSuperview()`, `viewWillMove(toWindow:)`, `displayTick()`.
- RL-052: `Task { }` inside `@MainActor` class inherits MainActor — `Process.waitUntilExit()` blocks main thread. Fix: `Task.detached(priority: .utility)`.
- RL-053: Mutating `@Observable` state from SwiftUI `body` = infinite re-render loop. Use `@ObservationIgnored` on async caches, pass pre-computed values to child views.

## Browser / WKWebView

- RL-048: harness-mcp DaemonClientActor default timeout=2s but WKWebView ops take 2–5s. Fix: `HarnessBrowserTools.send()` passes `timeout:35`.
- RL-055: `DateFormatter` init inside high-frequency main thread WKWebView callbacks blocks UI run loop. Offload to `DispatchQueue.global(qos: .utility)`.
- RL-056: High-frequency WKScriptMessageHandler (console.log) must batch writes — spawning one `DispatchQueue.global` work item per message saturates main thread. Throttle: collect into array, flush every 100ms with single async dispatch.
- RL-057: `PaneLifecycleManager` fast path (`!force && cached tabID → skip rebuild`) fires even when pane structure changes in-place (e.g. adding browser pane to existing tab). Guard: `cached !== paneContainer` — ensures fast path only applies to tab-switch restores, not structural mutations.
- RL-058: `NSSplitView.adjustSubviews()` after sidebar `isHidden` toggle redistributes ALL subview frames from scratch — triggers Metal surface layout cascade → 1-frame black flash. Use `setSidebarWidth() + split.layout()` instead. `adjustSubviews()` is only safe when no Metal terminal surfaces are in the subtree.
- RL-059: When an `NSScrollView`'s content looks visually shifted/clipped and each `layoutSubtreeIfNeeded()`/`displayIfNeeded()`/force-rebuild nudge only partially helps, stop adding nudges and check `documentView.isFlipped` first — a plain `NSView()` documentView anchors content to the bottom-left, not top-left. In a non-flipped view, `scroll(to: .zero)` scrolls to the BOTTOM, so a `scrollToTop()` built on it is a silent no-op. Symptom that many small timing fixes only partially converge is itself a signal to check for a structural (coordinate-space) cause instead of a timing one. See CASE-059.

## Architecture / Daemon

- RL-047: Split pane CWD must prefer `tab.worktreePath` over live process CWD. Agent CWD = repo root. Priority: `worktreePath → sourceCwd → tab.cwd`.
- RL-068: A "preserve local-only state across daemon sync" merge that checks "does the incoming snapshot lack this local-only thing" is not the same check as "did the daemon-side structure actually change" — if the local-only thing (e.g. a browser pane, inserted via `SplitPaneCoordinator.openBrowserPane`'s `applyLocalSnapshot`, deliberately bypassing daemon sync) can NEVER appear in the incoming snapshot by design, that condition is unconditionally true and the merge silently discards every real daemon-side structural change (new split, killed pane, ...) for as long as the local-only thing exists — with no error, no log line, because nothing failed; the successful daemon response was just thrown away on the next `applySnapshot`. **Recurring pattern, 2nd known occurrence** — the 1st (2026-06-15, `b76fb0dd`) fixed a *different* symptom of this exact same broken condition with a per-caller opt-out flag instead of fixing the condition itself, so it stayed broken for every other caller. Fix (2026-07-23): compare the incoming tree against the LOCAL tree with the local-only nodes stripped out, not "does incoming have any local-only nodes" — only preserve wholesale when the *stripped* structures match; otherwise adopt the incoming tree and re-attach the local-only nodes. Full incident history + "don't add another opt-out flag" guidance: `knowledge/bugs/browser-pane-local-daemon-merge.md`.

## Git / Process

- RL-060: `git revert --abort` (and `git merge --abort`) does `git reset --merge` back to the pre-operation HEAD — this discards working-tree edits to EVERY file that was part of the operation's index, not just the operation's own changes, including edits made after the operation started. Before aborting, back up (or `git stash`) any file touched since the in-progress operation began.

## Testing / Environment

- RL-061: Before diagnosing a visual/rendering diff between preview and production instances as a code bug, diff their per-instance `settings.json` (isolated preview state dirs start fresh and don't inherit the user's real customized settings — e.g. `minimumContrast`, `paletteHex` directly recolor terminal output). Confirm the underlying bytes/data are identical first (e.g. `script -q` raw capture) — if bytes match but rendering differs, check settings/config before renderer code. See CASE-060.
- RL-062: An `NSSplitView` arranged subview must stay in legacy autoresizing-mask sizing (`translatesAutoresizingMaskIntoConstraints = true`, the default) — NSSplitView positions/sizes arranged subviews via direct frame assignment (`setPosition`/`adjustSubviews`), not Auto Layout. A subview opted into `translatesAutoresizingMaskIntoConstraints = false` with no constraint defining its OWN width/position (only children pinned to its edges) is ambiguous to Auto Layout — any `layoutSubtreeIfNeeded()`/`layout()` call on it resolves the ambiguity to 0-width, silently wiping out whatever frame `setPosition` had just set. Symptom: `setSidebarWidth`'s computed `position` logs correctly, but `panel.frame` read moments later (after a `layoutSubtreeIfNeeded()` call) shows garbage (e.g. `(480, 0, 0, 760)` — width 0, wrong x). Fix: don't set `translatesAutoresizingMaskIntoConstraints = false` on the split's own arranged-subview container; only its *children* need Auto Layout constraints (pinned to the container's — frame-managed — bounds). Bug only manifested on the very first reveal of a previously-never-shown panel, since that's the first time this code path actually ran a `layoutSubtreeIfNeeded()` against the ambiguous view. See CASE-061.
- RL-065: Unfiltered `swift test` crashes with `bundleProxyForCurrentProcess is nil: mainBundle.bundleURL file:///Applications/Xcode.app/...` any time a code path calls `UNUserNotificationCenter.currentNotificationCenter()` (e.g. `DesktopNotifier.show` → `NotificationCoordinator.deliverAgentAlert`, itself reachable from `DaemonSyncService.applySnapshot`'s remote-notification fanout) — `UserNotifications` requires a real app bundle context that the `swift test` CLI process doesn't have (it runs from `Xcode.app/.../usr/bin/`, not `KouenPackageTests.xctest`'s own bundle). This is a **pre-existing test-environment limitation, not a code bug** — confirmed by reproducing it on clean `main` via `git stash` (2026-07-13, during P38 Phase A). It's nondeterministic/timing-dependent which test triggers it first, so the crash can appear to implicate whatever test happened to run adjacent to a snapshot-sync path. Workaround: use `swift test --filter <SomeOtherSuite>` for day-to-day iteration — the targeted-filter runs are reliable; only the full unfiltered run hits this. Real fix (not yet done) would be gating `DesktopNotifier.show` behind a check for a live bundle/test environment, or dependency-injecting the notification center so tests can stub it out.
- RL-063: An `async Task` that captures an `NSView` before an `await` (e.g. to anchor an `NSPopover`/menu once the await resolves) is unsafe if anything else in the app can rebuild the view tree containing that view while the await is in flight — a debounced `FSEventStream`-driven refresh, a `snapshotChanged` fanout rebuild, etc. Guard with `view.window != nil` (or `view.superview != nil`) immediately after the `await`, before using the view, and bail silently if it's been detached — don't assume the view you captured is still live just because you're holding a reference to it. Found via code review (not a live crash) in `GitPanelView.previewCommitDetail`: `applyState()` calls `historyStack.arrangedSubviews.forEach { $0.removeFromSuperview() }` on every git-history refresh, which could detach a commit card mid-`await runGit(["show", ...])`.

## Notifications / UserNotifications

- RL-064: A `UNUserNotificationCenterDelegate.willPresent` that unconditionally returns `[.banner, .sound]` silently ignores a per-notification sound toggle even when `content.sound` was correctly set to nil upstream — always derive presentation options from the content actually built (`content.sound != nil ? [.banner, .sound] : [.banner]`), don't hardcode them. Separately: `UNNotificationRequest(identifier: UUID().uuidString, ...)` with no `content.userInfo` means a delegate's `didReceive` (if one even exists) has nothing to route on — a notification needs its originating identity (e.g. `surfaceID`) written into `userInfo` at creation time, not reconstructed after the fact. Before assuming a toggle bug is deep, check `NotificationCenterProbe.isKnownBad` (or the app's equivalent probe flag) first — an AppleScript-fallback delivery path has no delegate at all, so "sound/click doesn't work" means something entirely different there than on the real API path. See CASE-063.
- RL-066: A "the click does nothing" bug can have a second, independent layer hiding behind the first: fixing *whether a handler fires* (event routing — gesture recognizer conflicts, `isTransparent` buttons swallowing target/action) does not verify *what the handler does once it fires* (its actual side effect / navigation target). `GitPanelView.cdToWorktree` blind-sendKeys `"cd <path>"` to whatever pane was currently focused instead of switching to the tab that tracks that path — this logic bug predated and was untouched by an earlier fix (CASE-038/1e7d3893) that only fixed the click-doesn't-fire symptom (confirmed via `git show <fixing-commit>`: the broken body moved verbatim). Before the earlier fix, the logic bug was invisible because the click never fired; after, it fired and hit the wrong target — same user-visible symptom, different root cause, read as "the bug came back" when it was actually never fixed. When closing a click-handling bug, verify the handler's *destination/effect*, not just that it executes — and when a user says a fixed bug "came back," check `git log -S`/`git show` on the original fixing commit before assuming regression; it may be an adjacent bug that was always there. See CASE-064.
