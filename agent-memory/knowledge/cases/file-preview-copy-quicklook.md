# File Preview: Copy/Find Broken, Worse for Markdown

**Summary.** File preview text couldn't be copied, and double-clicking a `.md` file to open it in the main editor tab appeared to do nothing useful. Three independent AppKit responder-chain gaps plus one inconsistency between the sidebar preview and main-tab editor's Quick-Look extension lists. Fixed across `SyntaxTextView.swift`, `FileViewerViewController.swift`, `FileEditorView.swift`, `FilePreviewCoordinator.swift`, `FileTreeSwiftUIView.swift`. Uncommitted as of this writeup, owner: session user + Claude Code.

**Symptom.** Reported in two passes:
1. "Can't copy" text shown in the sidebar file preview (`FileViewerViewController`).
2. Refined to: double-clicking a file (specifically `LANGUAGE.md`) to open it in the main editor tab area did nothing useful — no find, no copy, inconsistent with `.swift`/`.json` files which worked fine via the same double-click gesture.

**Root cause.** Four distinct gaps, found in order:

1. `SyntaxTextView` (`Apps/Kouen/Sources/KouenApp/UI/FileEditor/SyntaxTextView.swift`) is an `NSView` container wrapping an inner `NSTextView` (`SyntaxTextViewInner`). It's deliberately kept as first responder (not the inner text view) so its own `keyDown` can dispatch vi-mode commands. `NSView` has no `copy(_:)` — so when `SyntaxTextView` is first responder, `Edit > Copy` (wired with `target = nil` in `MainMenuBuilder.swift`, resolved via the responder chain) finds nothing that implements `copy(_:)` and silently no-ops, even though the inner text view's selection is visibly highlighted.
2. `FileEditorView` (`Apps/Kouen/Sources/KouenApp/UI/FileEditor/FileEditorView.swift`), the main-tab editor container, has the exact same shape: its own `keyDown` handles Cmd+F/Cmd+S directly, but it never implemented `copy(_:)` either.
3. Neither `FileViewerViewController.showText()` (sidebar preview) nor `FilePreviewCoordinator.loadActiveFileTab()` (main-tab editor) ever called `window.makeFirstResponder(...)` after loading a file — so immediately after opening a preview, first responder was whatever it was before (often a SwiftUI-hosted file-tree view), and neither Cmd+C nor Cmd+F had anywhere useful to land until the user made an unrelated extra click first.
4. The actual reported difference between `.md` and `.swift`/`.json`: `FileEditorView.load(path:)` (line 64) explicitly added `"md", "markdown"` to its Quick-Look-routed extension set (`qlExts`), which `FileViewerViewController`'s equivalent list (sidebar preview) does not have. `.swift`/`.json` never matched that set and always rendered through `SyntaxTextView` (the app's own syntax-highlighted, copyable, findable text view). `.md` files instead rendered through `QLPreviewView` (macOS's native Quick Look renderer) in the main tab — a completely different component that isn't wired into this app's `Edit` menu or `Cmd+F` at all.

**Why it produced the symptom.** Gap 4 alone fully explains the split behavior: `.swift`/`.json` always used `SyntaxTextView`, which (once gaps 1 and 3 are fixed) supports copy/find correctly. `.md` specifically detoured through Quick Look in the main tab, which supports neither — hence "double click .md preview doesn't work" while other file types were fine. Gaps 1–3 were real but secondary: they affected every file type, not just markdown, and were less visible because dragging to select text still visually worked (`NSTextView.mouseDown`'s internal event-tracking loop runs regardless of first-responder state) even when Cmd+C then failed.

**Fix.**
- `SyntaxTextView.copy(_:)` and `FileEditorView.copy(_:)` added, each forwarding to their inner `SyntaxTextView`/`textView.copy(sender)` — lets the responder chain resolve `Edit > Copy` correctly regardless of which container view currently holds first-responder status, without changing the existing (intentional) first-responder-stays-on-container design used for vi-mode/Cmd+F key dispatch.
- `FileViewerViewController.showText()` now calls `view.window?.makeFirstResponder(syntaxView)`; `FilePreviewCoordinator.loadActiveFileTab()` now calls `fileEditorView.window?.makeFirstResponder(fileEditorView)` — so a freshly opened preview is immediately interactive without a throwaway click first.
- Removed `"md", "markdown"` from `FileEditorView`'s `qlExts` set — markdown now renders through the same `SyntaxTextView` path as every other text file, matching the sidebar preview's own (unchanged) extension list.
- Also replaced the file-row double-click gesture in `FileTreeSwiftUIView.swift` (`.gesture(TapGesture(2).exclusively(before: TapGesture(1)))`) with two stacked `.onTapGesture(count:2)` / `.onTapGesture` modifiers — the pattern the folder rows in the same file already use successfully. This was investigated as a possible cause but confirmed (via temporary `NSLog` instrumentation) to already fire correctly either way; kept as a defensive simplification since `.exclusively` composition is a known-fragile SwiftUI/AppKit pattern and the file matched folder rows for consistency.
- Removed leftover `[DBG-cmdf]` `NSLog` instrumentation in `SyntaxTextView.mouseDown` that predated this session (found already in the working tree, from an earlier, inconclusive investigation of the same general area).

**How it was found.**
- Static source trace from `SyntaxTextView.swift` outward (`git diff`, `git blame`, `grep` for `makeFirstResponder`/`hitTest`/`copy(_:)`/gesture composition) — no debugger attach available for a running GUI app in this environment.
- Rejected hypothesis: first-responder handoff during `SyntaxTextView.mouseDown`'s forwarding to `textView.mouseDown(with:)` was unreliable. Investigated at length (matches pre-existing `[DBG-cmdf]` logging already in the tree from a prior session) but not the actual reported symptom — text selection via mouse-drag works regardless of first-responder state, since `NSTextView.mouseDown` runs its own internal event-tracking loop.
- Confirming experiment: rebuilt with `NSLog` probes on the file-row tap gestures, ran `log stream` filtered to the probe tag, had the user double-click a real file — `onTapGesture(count:2)` fired correctly, ruling out the gesture-composition hypothesis as the primary cause and pointing at the `qlExts` divergence instead once the user specified the affected file was `.md` and compared it against `.swift`/`.json` working.
- User's own manual repro (single-click preview → select → Cmd+C; double-click `.md` → nothing useful) was the deterministic signal throughout; no automated GUI test harness was available for interactive verification in this environment.

**Why it slipped through.** No test covered the Quick-Look-vs-text-view extension routing for either `FileViewerViewController` or `FileEditorView`, and the two components' extension lists were free to drift independently with no shared source of truth — nothing would have caught `FileEditorView` gaining `.md`/`.markdown` while `FileViewerViewController` didn't. Latent code: `FileEditorView`'s `qlExts` addition predates this session (unclear exact commit — `git blame`/`git log --follow` didn't surface it cleanly across an intervening module-rename commit) and had presumably been there long enough that only careful side-by-side testing surfaced the inconsistency.

**Validation.**
- New regression test `Tests/KouenAppTests/FileEditorViewQuickLookRoutingTests.swift` (3 cases: `.md` and `.swift` use the syntax view, `.pdf` still uses Quick Look) — passes (`swift test --filter FileEditorViewQuickLookRoutingTests`, 3/3, 0 failures).
- `swift build --product Kouen` clean after all edits including debug-instrumentation cleanup.
- Manually validated in a `make preview` (isolated preview app) build: user confirmed single-click sidebar preview copy works, and double-click `.md` open + copy now works ("ได้แล้ว").
- Not validated: other Quick-Look-routed extensions (`.pdf`, `.docx`, etc.) beyond the new test's `.pdf` case, and no coverage of the vi-mode keyDown path possibly interacting with the new `copy(_:)` forwards under every vi mode (normal/insert/visual).

**Action items / follow-ups.**
- None required for this fix to be complete. Optional future hardening: extract `qlExts`/Quick-Look-extension logic into one shared list `FileViewerViewController` and `FileEditorView` both read, so the two previews can't drift again the same way — not done here to keep this fix's diff minimal and scoped to the reported bug.
